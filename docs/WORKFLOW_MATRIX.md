# HXForge Workflow and Capability Matrix — v0.1 Draft

## 1. Purpose

This document defines the minimum information required for each workflow, the capability boundary for v0.1, and the status that must be returned when a request is incomplete or unsupported.

## 2. Workflow definitions

| Workflow | User intent | Required result |
|---|---|---|
| Screening | Compare exchanger families before geometry is selected | ranked technically plausible families, reasons, blockers and missing information |
| Sizing | Generate geometry that satisfies a thermal duty and constraints | one or more manufacturable candidates with thermal/hydraulic results |
| Rating | Evaluate a specified geometry at a specified operating condition | duty, outlet states, pressure drops, margins, warnings and provenance |
| Comparison | Compare completed candidates | normalized technical, risk and cost comparison |
| Report | Package an existing run | report containing inputs, assumptions, warnings, limitations and trace |

## 3. Minimum input matrix

| Input group | Screening | Sizing | Rating |
|---|---:|---:|---:|
| workflow identifier | required | required | required |
| exchanger family | optional / auto | required or auto-screened | required |
| hot and cold fluid identifiers | required | required | required |
| inlet temperature and pressure on both sides | required | required | required |
| mass flow on both sides | normally required | required unless an approved solved-variable mode exists | required |
| target duty or at least one outlet temperature | required | required | optional when geometry and inlet states define a solvable rating problem |
| allowable pressure drop on both sides | recommended | required for constrained sizing | optional but required for compliance-to-limit checks |
| fouling resistance | optional for early screening | required or explicitly zero with structured source (see DEC-017) | required or explicitly zero with structured source (see DEC-017) |
| material constraints | recommended | conditional | conditional |
| design pressure and design temperature | optional for pure thermal screening | required before mechanical checks | required before mechanical checks |
| geometry schema | not used | generated by solver | required and versioned |
| maintenance / cleaning constraints | recommended | conditional | conditional |
| footprint limits | optional | conditional | not required to calculate, required to check fit |

## 4. Specification closure rules

| Condition | Required behavior |
|---|---|
| No duty and no outlet temperature supplied | `BLOCKED` as under-specified |
| Duty and both outlet temperatures supplied but inconsistent | `BLOCKED` with energy-balance inconsistency |
| One outlet temperature supplied with valid inlet states and flows | solver may calculate duty and the other outlet when the service is supported |
| Target duty supplied with valid inlet states and flows | solver may calculate outlet states when the service is supported |
| Unknown or unsupported geometry fields in rating | reject the request; do not ignore fields |
| Area margin applied in sizing | candidate accepted when `actual_area >= required_area × (1 + margin)`; see DEC-015 |
| Area margin in rating | reported as `area_margin` and `duty_margin`; does not modify the calculation; see DEC-015 |
| Missing engineering value that has no approved default | `BLOCKED`; never guess |
| Requested capability exists in architecture but has no validated implementation | `NOT_IMPLEMENTED` |
| Mass flow rate is zero or negative | `BLOCKED` with validation error |
| Operating temperature exceeds design temperature | `WARNING` with explicit note; does not block thermal calculation but blocks mechanical conclusions |
| Property backend cannot identify or evaluate the requested fluid | `BLOCKED` with property error |
| Ambiguous phase state (e.g., near-saturation with auto phase hint) | `BLOCKED` unless validated phase-determination logic is available; otherwise require explicit `phase_hint` |
| Target duty and both outlet temperatures all provided (over-specified) | verify energy-balance consistency; proceed with `WARNING` if consistent, `BLOCKED` if inconsistent |

## 7. Result states

### 7.1 Status (execution state)

Status tracks the workflow execution position. A run progresses through the following states:

`DRAFT → INPUT_VALIDATED → THERMAL_SERVICE_RESOLVED → TECHNOLOGIES_SCREENED → CANDIDATES_GENERATED → CANDIDATES_RATED → ENGINEERING_CHECKED → COSTED → VERIFIED → REPORT_READY`

A run may terminate at any point with a terminal status:

| Terminal status | Trigger |
|---|---|
| `BLOCKED` | Input, safety, applicability, property or specification closure failure |
| `NOT_IMPLEMENTED` | Requested capability exists in architecture but has no validated implementation |
| `NON_CONVERGED` | Iterative solver reached maximum iterations without meeting tolerances (see DEC-014) |

### 7.2 Verification level (result maturity)

Verification level describes the engineering confidence of the result, independent of the execution status:

| Level | Meaning |
|---|---|
| `PRELIMINARY` | Calculation completed; requires engineering review |
| `REVIEW_REQUIRED` | Result exists but assumptions or warnings need approval |
| `VERIFIED` | Passes approved benchmark and review rules |
| `N/A` | Not applicable when status is a terminal state (BLOCKED, NOT_IMPLEMENTED, NON_CONVERGED) |

The Agent must not advance the status merely to satisfy a user request.
